# Требования к библиотеке LRU Cache

## 1. Общее описание
Целью проекта является разработка высокопроизводительной, потокобезопасной библиотеки для реализации кэша с политикой вытеснения Least Recently Used (LRU) на языке Rust. Библиотека должна поддерживать различные режимы управления объемом данных и временем их жизни, а также обеспечивать возможность сохранения и восстановления состояния кэша.

## 2. Функциональные требования

### 2.1. Режимы работы кэша
Библиотека должна поддерживать три режима функционирования:

1.  **Режим ограничения по емкости (Capacity-only mode):**
    *   Устанавливается максимальное количество элементов ($N$), которые могут одновременно находиться в кэше.
    *   При попытке вставки нового элемента, если кэш заполнен, наиболее старый элемент (тот, к которому дольше всего не обращались) должен быть удален.
    *   Любое обращение к элементу (чтение или обновление) должно помечать его как "недавно использованный", обновляя его позицию в очереди вытеснения.

2.  **Режим ограничения по времени жизни (TTL-only mode):**
    *   Для каждого элемента устанавливается максимальное время жизни (Time-To-Live, TTL).
    *   При каждом обращении к элементу (чтение/обновление) счетчик времени жизни сбрасывается, и элемент получает полный новый период TTL.
    *   Элементы, срок жизни которых истек, должны считаться невалидными и удаляться (или не возвращаться при запросе).
    *   В этом режиме ограничение на общее количество элементов не применяется.

3.  **Комбинированный режим (Hybrid mode):**
    *   Одновременно применяются оба правила: ограничение по количеству элементов и ограничение по времени жизни каждого элемента.
    *   Удаление происходит либо при достижении лимита емкости (по LRU), либо при истечении TTL.

### 2.2. Потокобезопасность
*   Кэш должен быть безопасным для использования в многопоточной среде (`Send + Sync`).
*   Должна быть обеспечена корректная синхронизация при одновременном чтении, записи и обновлении времени использования/жизни элементов.

### 2.3. Персистентность (Persistence)
*   Библиотека должна предоставлять механизмы для:
    *   **Сохранения (Save):** Сериализация текущего состояния кэша (ключи, значения, метаданные времени использования) и запись на диск.
    *   **Восстановления (Load):** Чтение данных с диска и восстановление структуры кэша с сохранением актуальности времени жизни элементов (если это возможно технически).
*   **Форматы данных:** Должна поддерживать как минимум текстовый (JSON) и бинарный форматы сериализации.

## 3. Интерфейс (API)
Интерфейс должен быть:
*   Простым и интуитивно понятным.
*   Типизированным (использование Generics для ключей и значений).
*   Документированным (Rustdoc).

Основные ожидаемые методы:
*   `get(key)` — получить значение.
*   `put(key, value)` — добавить или обновить значение.
*   `remove(key)` — удалить элемент.
*   `clear()` — очистить кэш.
*   `contains(key)` — проверить наличие ключа в кэше.
*   `len()` — вернуть текущее количество элементов в кэше.
*   `save(path)` — сохранить состояние (через адаптер).
*   `load(path)` — восстановить состояние (через адаптер).

## 4. Нефункциональные требования
*   Высокая производительность операций вставки и поиска.
*   Минимальные накладные расходы на память.
*   Надежность при работе с файловой системой.
